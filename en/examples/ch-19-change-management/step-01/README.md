# example_19_01

**Chapter 19: Change management** · step 01 (runnable changesets demo)

Minimal setup with `@changesets/cli` + a fake `@ds/react` package + the
exact changeset from the book. Run `npm install && npm run version` and
changesets writes a real `CHANGELOG.md` and bumps the package version from
2.2.0 to 3.0.0 (because the changeset declares `major`).

## How to run it

```bash
cd en/examples/ch-19-change-management/step-01
npm install
npm run version
```

After `version`:

- `packages/react/package.json` jumps to `"version": "3.0.0"`.
- A new `packages/react/CHANGELOG.md` is created with a `## 3.0.0` entry
  containing the changeset text.
- The `.changeset/renamed-button-variant-prop.md` file is deleted
  (changesets consumes it on bump).

To reset to the initial state, `git checkout -- .` before experimenting
again.

## The file that matters

```
.changeset/
  renamed-button-variant-prop.md  ← example_19_01 verbatim from the book
  config.json                     ← changesets config, generated by init
  README.md                       ← instructions for devs
```

The changeset itself is three effective lines:

```md
---
"@ds/react": major
---

Renamed Button.intent to Button.variant...
```

The frontmatter declares the package and the bump type (major). The body is
the changelog entry the final reader will see.

## Trying the full PR flow

With `npm` or `pnpm` installed, the real flow reproduces like this:

1. **Edit package code.** Simulate a breaking change: add a line to
   `packages/react/src/index.ts`.
2. **Create a new changeset.** `npm run changeset`. The CLI asks, you
   answer, it writes a second file in `.changeset/`.
3. **Commit the code + the changeset.** In a real repo this would be the
   PR.
4. **Publish.** `npm run version` bumps versions and regenerates the
   changelog; `npm run publish` would push to npm (here it skips because
   `private: true`).

## Things worth tweaking

- Change the bump type in the frontmatter to `minor` and run `version`
  again. The version jumps to 2.3.0 instead of 3.0.0. That one-word change
  is the difference between "consumers migrate" and "consumers upgrade
  without looking".
- Create a second dummy package (`@ds/tokens`) and a changeset that
  affects both. You'll see changesets bump both respecting the type per
  package.
